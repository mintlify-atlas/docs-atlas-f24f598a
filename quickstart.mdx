---
title: "Quick Start"
description: "Get up and running with Bencher in minutes."
---

<Steps>
  <Step title="Add Bencher via CMake FetchContent">
    Add the following to your `CMakeLists.txt`:

    ```cmake CMakeLists.txt
    include(FetchContent)
    FetchContent_Declare(
        bencher
        GIT_REPOSITORY https://github.com/stephenberry/bencher.git
        GIT_TAG main
    )
    FetchContent_MakeAvailable(bencher)

    target_link_libraries(your_target PRIVATE bencher::bencher)
    ```

    This fetches Bencher at configure time and makes the `bencher::bencher` target available to link against.
  </Step>

  <Step title="Write your first benchmark">
    Create a `benchmark.cpp` file with a stage, one or more `run` calls, and a `print_results` call:

    ```cpp benchmark.cpp
    #include "bencher/bencher.hpp"
    #include "bencher/diagnostics.hpp"

    int main()
    {
       bencher::stage stage{"My Benchmarks"};

       stage.run("Algorithm A", [] {
          // Your code to benchmark
          // ...
          return bytes_processed; // Return the number of bytes processed
       });

       stage.run("Algorithm B", [] {
          // Another implementation
          // ...
          return bytes_processed;
       });

       bencher::print_results(stage);

       return 0;
    }
    ```

    The lambda returns the number of bytes processed. Bencher uses this to compute throughput in MB/s. If throughput is not meaningful for your workload, you can return `void` — timing metrics are still collected.
  </Step>

  <Step title="Build and run">
    Configure and build with CMake, then run the resulting binary:

    ```bash
    cmake -B build -DCMAKE_BUILD_TYPE=Release
    cmake --build build
    ./build/benchmark
    ```

    Use `Release` mode. Debug builds disable optimizations and will produce artificially slow results.
  </Step>

  <Step title="Understand the output">
    After all benchmarks in a stage complete, Bencher prints a table of metrics for each subject followed by a comparison:

    ```
    Performance Metrics for: JSON benchmarks
    ----------------------------------------------------
     - JSON Read -
    Bytes Processed                         :        722
    Throughput (MB/s)                        :       1847
    Throughput MAD (±%)                      :       1.2
    Instructions per Execution              :      45023
    Instructions per Cycle                  :       2.31
    Cycles per Execution                    :      19487
    Frequency (GHz)                         :       3.21
    Total Iterations                        :         47
    ====================================================
    ```

    | Field | What it tells you |
    |-------|-------------------|
    | **Throughput (MB/s)** | How fast data is processed. Higher is better. |
    | **Throughput MAD (±%)** | Variability of throughput across iterations. Under 3% is typical; above 5% suggests noise. |
    | **Instructions per Cycle (IPC)** | CPU efficiency. Values above 1.5 are typical; below 1 often indicates memory stalls. |
    | **Cycles per Execution** | Raw CPU cycles for one benchmark run. |
    | **Frequency (GHz)** | Estimated clock speed, derived from cycles and wall time. |
    | **Total Iterations** | How many iterations Bencher ran before the confidence interval converged. |

    When a stage has more than one subject, Bencher also prints a bar chart comparison:

    ```
    JSON Write is 23.5% faster than JSON Read
    ====================================================

    ████████████████████████████████████████│ JSON Write (2284)
    ████████████████████████████████▍       │ JSON Read (1847)
    ```
  </Step>
</Steps>

<Note>
  Bencher runs a 1-second warmup automatically before the first benchmark in each stage. This lets the CPU reach its steady-state frequency before measurements begin, which improves reproducibility. You do not need to add a warmup yourself.
</Note>
